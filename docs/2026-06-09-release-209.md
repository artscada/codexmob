# 2026-06-09: изменения в Codex Android RU и релиз 209

Ниже зафиксированы изменения, сделанные за день, с упором на shell-маршрут, настройки безопасности, HTTP/API и запуск на устройстве `209`.

## Ключевая цель дня

- Вернуть и стабилизировать shell-путь в `codex-android`.
- Сделать поведение понятнее в UI и в runtime-логике.
- Проверить, что устройство `209` реально исполняет root shell через приложение и HTTP API.
- Подготовить релизный APK и вынести его как артефакт для GitHub Release с пометкой `209`.

## Что поменялось в коде

### 1) `SecurityPolicy`: добавлен режим `shell_only`

Файл: `app/src/main/java/com/codex/android/security/SecurityPolicy.kt`

Что сделано:

- Добавлен новый уровень `shell_only`.
- Изменена логика `isShellAllowed()`:
  - раньше shell был доступен только в `full`;
  - теперь shell доступен в `shell_only` и `full`.
- Изменена логика `isFullFileAccessAllowed()`:
  - `standard` и `full` оставляют доступ к файлам по всей системе;
  - `safe` и `shell_only` ограничивают файловые инструменты рабочим каталогом.
- Обновлены тексты ошибок:
  - shell-denied сообщение теперь подсказывает `仅 Shell / 完全` как допустимые режимы.
- Обновлены комментарии:
  - пояснено, что file sandbox относится к файловым инструментам, а не к shell как таковому.

Итог:

- `shell_only` стал отдельным режимом, а не скрытым обходом.
- Shell можно включать без перехода в полный файловый доступ.

### 2) Настройки: добавлен пункт `Только shell`

Файл: `app/src/main/java/com/codex/android/ui/settings/CodexSettingsScreen.kt`

Что сделано:

- В секцию `Уровень безопасности` добавлен четвёртый radio-пункт:
  - `Только shell`
- Для него используется:
  - `security_level = "shell_only"`
- В UI описание говорит, что:
  - shell включён;
  - файловые инструменты остаются в рабочем каталоге.

Итог:

- Теперь есть отдельный визуальный выбор для shell-only режима.
- Это убрало необходимость временно прятать поведение в `full`.

### 3) `CodexLocalExecBridge`: прямой shell стал надёжнее

Файл: `app/src/main/java/com/codex/android/bridge/CodexLocalExecBridge.kt`

Что сделано:

- `sendPrompt(...)` получил параметр `freshChat: Boolean = false`.
- При `freshChat = true` создаётся новая локальная chat-сессия, а не переиспользуется старая.
- Убрана шумная строка статуса:
  - `Локальный Codex выполняет запрос...`
- Добавлен парсер явных shell-запросов:
  - `Android shell: ...`
  - `root shell ...`
  - `adb shell ...`
  - `shell: ...`
  - `run command ...`
- Добавлены правила извлечения shell-команды из свободной формулировки.
- Добавлена спец-логика для `service.d` и `.sh`:
  - если пользователь просит прочитать shell-скрипт из `service.d`, мост переводит это в `cat /data/adb/service.d/<file>.sh`.

Итог:

- Shell-команды стали распознаваться устойчивее.
- Запросы вида “используя root shell прочитай файл…” перестали зависеть от очень точной формулировки.

### 4) `CodexApiBridge`: поддержка fresh chat

Файл: `app/src/main/java/com/codex/android/bridge/CodexApiBridge.kt`

Что сделано:

- `sendPrompt(...)` получил параметр `freshChat`.
- Если `freshChat = true`, создаётся новая chat-сессия под API-режим.

Итог:

- Поведение API-ветки стало согласованным с локальным bridge и HTTP-обвязкой.

### 5) `CodexExternalHttpServer`: улучшен маршрут `/api/run-task`

Файл: `app/src/main/java/com/codex/android/integrations/http/CodexExternalHttpServer.kt`

Что сделано:

- Для `task_id`-запросов принудительно создаётся свежий чат.
- `executeWithApiBridge(...)` и `executeWithLocalBridge(...)` получили параметр `freshChat`.
- `bridge.sendPrompt(...)` теперь вызывается с этим флагом.
- Улучшено чтение `postData`:
  - сервер пытается читать тело как путь к временному файлу, если NanoHTTPD передал именно file path.

Итог:

- `/api/run-task` стал надёжнее для task-based сценариев.
- Избавились от части случаев, когда запрос оказывался “в старом диалоге”.

### 6) `AnyclawManager`: встроенный пакет shell стал “Root Shell”

Файл: `app/src/main/java/com/codex/android/codex/AnyclawManager.kt`

Что сделано:

- `android-shell` переименован в `Root Shell`.
- Описание и текст tool’а обновлены с `Android Shell` на `Root Shell`.
- Встроенные пакеты теперь гарантированно помечаются как `isInstalled = true`.
- Логика загрузки/добавления встроенных пакетов стала чище:
  - не теряются при повторной инициализации;
  - не теряют флаг установленности.

Итог:

- В UI и MCP теперь используется более точное название.
- Встроенные пакеты ведут себя стабильнее после перезапуска.

### 7) `CodexMCPBridge`: обновлено название shell tool

Файл: `app/src/main/java/com/codex/android/provider/CodexMCPBridge.kt`

Что сделано:

- Текст builtin tool `android_shell` изменён на `执行 Root Shell 命令`.

Итог:

- MCP-слой больше не вводит в заблуждение названием.

### 8) `AndroidUtils.js`: исправлен вызов shell helper

Файл: `app/src/main/assets/js/AndroidUtils.js`

Что сделано:

- `Tools.system.shell(...)` заменён на `Tools.System.shell(...)`.

Итог:

- Исправлен вызов API с правильным регистром.
- Это важно для JavaScript-слоя, который дергает shell helper.

- Изменения не только внесены в код, но и подтверждены на железе `209`.

- Если понадобится вернуться к прежней схеме, можно убрать `shell_only` из UI и оставить только `safe / standard / full`.
